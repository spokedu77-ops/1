# 스포키듀 Pro ↔ 센터 커리큘럼 동기화 — 구체화된 권장 방향

현재 구현 기준: [`import-center`](app/api/spokedu-pro/programs/import-center/route.ts)는 `curriculum` → `spokedu_pro_programs` **일방향 덮어쓰기**이며, Pro 전용 수정과의 관계를 팀에서 명문화하는 것이 좋다.

---

## 1. 원본(Source of truth) 정책 — 선택지 3가지

### A. 커리큘럼 단일 원본 (가장 단순, 현재 코드와 가장 잘 맞음)

| 영역 | 원본 | Pro에서 |
|------|------|-----------|
| 제목, 영상 URL, 체크리스트, 교구(문자열), 활동 단계, 팁 | **센터 커리큘럼**만 수정 | 보기 전용 또는 “커리큘럼에서 수정” 안내 |
| 분류(신체 기능·활동 테마·인원 등) | 동기화 시 **고정 매핑**으로 덮어씀 | Pro에서 고쳐도 **다음 동기화에 소실** — **건드리지 않는 것**을 권장 |

- **장점**: 운영 규칙이 단순하고, 동기화 결과가 항상 예측 가능하다.
- **단점**: Pro만의 “수업용 미세 조정”은 불가에 가깝다.

**추천 대상**: 커리큘럼 팀이 콘텐츠를 일괄 관리하고, Pro는 구독 노출용인 조직.

#### A안 + 현재 코드: “공통만 바뀌고, 구독용 태그는 유지?”에 대한 정확한 답

**직감과 조금 다를 수 있음.** 동기화는 `curriculum`에 없는 “개념”만 남는 게 아니라, **Supabase `update`에 실리는 컬럼만** 커리큘럼(또는 고정 매핑) 값으로 덮어쓴다. **payload에 없는 컬럼·별도 저장소**는 손대지 않는다.

- **동기화 한 번 돌릴 때마다 덮어쓰는 것(공통 본문 + 고정 분류에 해당)**  
  `title`, `video_url`, `checklist`, `equipment`, `activity_method`, `activity_tip`, `function_type`, `main_theme`, `group_size` 등 — [`import-center` payload](app/api/spokedu-pro/programs/import-center/route.ts)에 들어 있는 필드.  
  여기서 말하는 **교구 문자열(`equipment`)** 도 커리큘럼 `equipment`에서 온다. 프로그램 뱅크 카드에 쓰는 **활동 테마·신체 기능·교구 요약**도 이 DB 값에서 나오므로, **“커리큘럼에 필드가 없다”고 해서 이 태그들이 동기화에서 제외되는 것은 아니다**(값은 커리큘럼·고정 매핑 기준으로 맞춰짐).

- **그대로 남을 수 있는 것(구독·Pro 전용에 가깝게 쓰기 좋은 쪽)**  
  - **`spokedu_pro_programs`에 있으나 동기화 payload에 안 넣는 컬럼** (예: `function_types` 배열 등 — 현재 import-center는 넣지 않음)  
  - **`program_details` 등 콘텐츠 블록**에만 저장한 제목/영상/메모 — import-center는 이 블록을 갱신하지 않음 (단, 화면이 DB와 블록을 어떻게 합치는지에 따라 보이는 값은 달라질 수 있음)

**정리 (A안 채택 시 팀에 전달할 문장)**  
“**커리큘럼과 맞춰 공통으로 쓰는 본문·URL·체크리스트·교구(리스트)·단계·팁, 그리고 동기화 시 고정되는 분류 한 줄**은 동기화 때마다 Pro DB에서도 그쪽으로 맞춰진다.  
**커리큘럼 테이블에 ‘태그’ 칼럼이 없다**고 해서, **프로그램 뱅크에 보이는 테마/기능/교구 표시가 자동으로 ‘Pro만의 값’으로 남지는 않는다.**  
구독 전용으로 오래 유지하려면 **동기화가 건드리지 않는 저장소(별도 컬럼·블록)**에 두거나, 정책 B처럼 **동기화 규칙을 코드에서 바꾼다**.”

---

### B. 이원화 — “본문은 커리큘럼, 분류·메타는 Pro” (제품 요구가 있을 때)

| 영역 | 원본 | 동기화 시 |
|------|------|-----------|
| 제목·URL·체크리스트·교구·단계·팁 | 커리큘럼 | **항상** 커리큘럼 값으로 갱신 |
| `main_theme`, `function_type` / `function_types`, (선택) `group_size` 등 | **Pro** | 동기화 시 **payload에서 제외**하거나, “Pro에서 수정됨” 플래그가 있을 때만 스킵하는 **코드 변경** 필요 |

- **장점**: 구독 화면의 필터/카드 메타는 센터와 무관하게 다듬을 수 있다.
- **단점**: 구현·테스트·문서화 비용이 있다. 잘못 설계하면 다시 “꼬임”이 생긴다.

**추천 대상**: 커리큘럼은 “공식 영상·문구”만 맞추고, 분류는 센터별/기능별로 Pro에서 조정해야 하는 경우.

---

### C. Pro 단일 원본 (비권장에 가깝)

- 커리큘럼은 초기 시드만 하고 이후에는 Pro만 진실로 둔다 → **현재 `import-center`와 정면 충돌**한다.
- 동기화를 아예 다른 의미(예: 신규 행만 삽입, 기존 행은 URL만 양방향 비교)로 바꿔야 해서 범위가 크다.

**추천**: 특별한 사유 없이는 채택하지 않는 편이 안전하다.

## 2. 추천 기본안 (대부분 팀에 맞는 절충)

1. **문서/온보딩 한 줄**  
   “**센터 커리큘럼에서 고치는 것**: 제목·영상·체크리스트·교구·활동 단계·팁.  
   **동기화(`가져오기`) 후에는 Pro에 있던 해당 값은 커리큘럼 내용으로 덮어씁니다.**  
   분류(테마·신체 기능 등)는 동기화 때마다 기본값으로 맞춰지므로, Pro에서만 유지하려면 정책 B로 개발 요청이 필요합니다.”

2. **역할 분리**  
   - **커리큘럼 담당**: URL·본문 일괄 수정.  
   - **Pro/구독 담당**: 동기화 실행 타이밍(수업 전·배포 전) 합의.  
   - **Pro에서 임시 수정**했다면: “다음 동기화 전에 커리큘럼에 반영할지” 체크리스트에 넣기.

3. **동기화 직전 UI (선택, 추후 구현)**  
   - Admin `가져오기` 버튼 근처에 **비차단 경고** 1줄:  
     “연결된 프로그램의 제목·영상·체크리스트·교구 등이 커리큘럼 기준으로 갱신됩니다.”  
   - 확장 시: 마지막 동기화 시각 / `updated` 건수 요약.

---

## 3. 동기화 이후 “꼬임” 방지 체크리스트 (운영)

- [ ] 같은 커리큘럼에 대해 **Pro 중복 행**(옛 URL 등)이 없는지 — 필요 시 `hardDelete` 가져오기로 정리(이미 Admin 플로우에 있음).
- [ ] `program_details` 등 **콘텐츠 블록**에 예전 `videoUrl`이 남아 덮어쓰지 않는지(드문 케이스).
- [ ] `function_type` vs `function_types` 불일치가 없는지(동기화가 단일 필드만 넣는 경우).

## 4. 다음 단계(제품/개발) — 원할 때만

| 단계 | 내용 |
|------|------|
| P0 | 팀 규칙을 **A 또는 B** 중 하나로 확정하고, 위 1~3을 내부 문서/슬랙에 고정 |
| P1 | Admin 동기화 버튼에 **경고 문구 1줄** (코드 소량) |

## 5. 한 줄 요약

**지금 코드는 “커리큘럼이 본문의 원본”에 가깝고, 동기화는 그 본문 + 고정 분류를 Pro에 다시 박는다.**  
Pro에서만 유지하고 싶은 것이 있으면 **팀에서 A vs B를 먼저 정한 뒤**, B일 때만 개발 이슈로 분리하는 것이 좋다.

## 6. 대안 총정리 — “그래서 방안이 뭐냐”에 대한 실행 가능한 선택지

아래는 **코드 변경 없음 → 작음 → 큼** 순으로 대안을 나열한 것이다. 팀은 **하나를 골라 고정**하면 꼬임이 줄어든다.

### 대안 0 — 운영만 (코드 그대로, A안 유지)

- **내용**: 커리큘럼이 본문 원본, Pro는 동기화 후에도 “공통 필드”는 커리큘럼과 같다고 가정. 구독 전용 문구는 **동기화가 안 건드리는 곳**에만 둔다(`program_details` 등).
- **장점**: 즉시 적용, 비용 없음.
- **한계**: 카드에 쓰는 테마·기능·교구 문자열은 **동기화와 함께** 커리큘럼 쪽으로 맞춰짐(이미 서술한 대로).

### 대안 1 — UI/문서만 (P1)

- **내용**: Admin `센터 커리큘럼 가져오기` 버튼 옆에 **한 줄 경고** + 내부 위키에 “덮어쓰는 필드 목록” 고정.
- **장점**: 실수로 기대한 “태그 유지”를 줄임.
- **한계**: 동작 자체는 변하지 않음.

### 대안 2 — 동기화 payload 축소 (“본문만 커리큘럼”, B의 최소형)

- **내용**: [`import-center`](app/api/spokedu-pro/programs/import-center/route.ts)의 `update`/`insert`에서 **`main_theme`, `function_type`, `group_size`(및 필요 시 `function_types`)를 payload에서 제거**한다. 그러면 동기화는 **제목·URL·체크리스트·교구·단계·팁**만 커리큘럼과 맞추고, **분류·복수 기능 타입은 Pro에서 마지막으로 저장한 값이 유지**된다.
- **장점**: 구현 범위가 비교적 작고, “구독용으로 고친 분류/태그” 요구와 잘 맞는다.
- **리스크**: 신규 행 insert 시 분류 기본값을 어디서 줄지(마이그레이션·첫 저장 규칙) 한 번 정해야 함. 커리큘럼만으로 “펑셔널 무브 기본 3종”을 강제하던 현재 정책과는 달라짐.

### 대안 3 — 플래그 기반 부분 덮어쓰기 (B의 안전한 형태)

- **내용**: `spokedu_pro_programs`에 예: `pro_meta_locked` 또는 컬럼별 타임스탬프. **Pro에서 분류를 저장하면 잠금**, import-center는 **잠긴 필드는 스킵**, 나머지만 커리큘럼으로 갱신. (또는 “잠금 해제” 버튼으로만 커리큘럼 값으로 되돌리기.)
- **장점**: “이 프로그램만 커리큘럼 따라가기 / 이건 Pro 고정”을 섞을 수 있음.
- **리스크**: UI·권한·엣지 케이스 설계 필요, 구현량 중간~큼.

### 대안 4 — 커리큘럼 스키마 확장 (데이터 원본을 하나로 유지하면서 태그도 공통화)

- **내용**: `curriculum`에 **뱅크용 태그/분류 컬럼**(또는 JSON)을 추가하고, 어드민 커리큘럼 UI에서 편집. import-center가 그 값을 그대로 `main_theme` / `equipment` 문자열 등에 매핑.
- **장점**: A안 철학 유지 — “진짜 원본은 항상 커리큘럼 한 곳”.
- **리스크**: DB 마이그레이션 + 커리큘럼 UI + 기존 데이터 백필 작업이 큼.

### 대안 5 — Pro 전용 오버레이 컬럼/테이블 (표시만 분리)

- **내용**: 예: `bank_overlay` JSONb (`display_theme`, `display_functions`, …`) **import-center는 절대 갱신하지 않음**. 프로그램 뱅크 카드/필터는 **overlay 있으면 overlay, 없으면 DB 기본** 컬럼.
- **장점**: 커리큘럼 동기화와 **완전 분리**된 구독 표시를 명확히 할 수 있음.
- **리스크**: 마이그레이션 + Library/Drawer/API 읽기 규칙 전부 합의 필요(구현 큼).

---

### 추천 조합 (실무)

| 상황 | 추천 |
|------|------|
| 당장 꼬임만 줄이고 싶다 | **대안 0 + 1** |
| “영상·본문은 커리큘럼, 뱅크 태그/분류는 Pro 유지”가 제품 요구다 | **대안 2**를 1차로 검토 → 필요하면 **대안 3**으로 고도화 |
| 장기적으로 데이터 원본을 한 곳에 두고 싶다 | **대안 4** (또는 4와 2 병행 로드맵) |

위 표 중 **하나를 제품에 공식 채택**하고, 나머지는 “하지 않는다”고 박아 두는 것이 대안의 핵이다.

## 7. 확정 스펙(요청): 커리큘럼 → Pro 동기화는 아래만

**동기화(`import-center`) UPDATE 시 커리큘럼에서 가져와 덮어쓸 필드만** 다음으로 한정한다는 뜻으로 이해한다.

| 커리큘럼(센터) | Pro DB (`spokedu_pro_programs`) |
|----------------|----------------------------------|
| URL (YouTube / Shorts) | `video_url` |
| Checklist (엔터 구분) | `checklist` (`check_list` → join) |
| Equipment (엔터 구분) | `equipment` (`equipment` 배열 → join) |
| Activity Steps (엔터 구분) | `activity_method` (`steps` → join) |
| Expert Tip | `activity_tip` (`expert_tip`) |

**동기화 시 더 이상 덮어쓰지 않고 Pro에서 독자 유지**하는 예시:

- `title` (프로그램명)
- `function_type`, `function_types`, `main_theme`, `group_size`
- 구독/어드민에서 따로 두는 메타(향후 오버레이 컬럼 등)

### 구현 시 유의 (반드시 정할 것)

1. **`title`을 동기화에서 제외할지**  
   - 제외하면: 커리큘럼에서 제목만 바꿔도 Pro 뱅크 제목은 **안 바뀜** (독립).  
   - 포함하면: 아래 5개만 동기화라는 요청과 **모순**되므로, **제외**가 이 스펙과 맞다.

2. **`source_center_curriculum_id` / `is_published` / SUB 처리**  
   - 행 매칭·비공개 처리에는 여전히 필요할 수 있음.  
   - “본문만”이 아니라 **링크·노출 제어**용으로 `UPDATE`에 **소량 필드 추가**할지(예: `source_center_curriculum_id`만 갱신)는 구현 단계에서 한 번 더 박는 것이 안전하다.

3. **INSERT(커리큘럼에만 있고 Pro에 없는 신규 행)**  
   - 첫 생성 시 `title`·분류 기본값을 **어디서 채울지**(커리큘럼 `title`을 한 번만 넣고 이후엔 Pro 독립 등) 규칙을 짧게 정한다.

이 절은 **대안 2보다 더 좁은 동기화 범위**이며, 구현 시 [`import-center`](app/api/spokedu-pro/programs/import-center/route.ts)의 `payload`를 위 표 + (필요 시) 링크 필드만 남기면 된다.

**구현 반영(코드)**: `import-center`의 **UPDATE**는 위 본문 5종 + `source_center_curriculum_id`(행 연결)만 갱신한다. **INSERT**(신규)는 제목·본문 5종·기본 분류·`is_published`까지 한 번에 넣는다.
