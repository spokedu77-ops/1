# 스포키듀 구독 — 앱 전문가 시각 고도화 로드맵

방향성(1→2→3)과 체크리스트를 전제로, **기능 고도화·UX·안정성**을 단계별로 풀어낸 상세 계획.  
각 단계는 “무엇을, 어디서, 어떻게 검증할지”까지 명시한다.

---

## 0. 현재 상태와 갭

| 영역 | 현재 | 갭(고도화 포인트) |
|------|------|-------------------|
| **1번(주간 자료)** | 대시보드 Row1 “이번 주 테마” 4개 노출, 큐레이션 편집 가능 | “이번 주” 갱신 주기·신뢰감 부족, 빈/에러 상태 미정비, 144개 풀과의 연결 불명확 |
| **2번(수업 보조)** | 술래/팀 나누기 구현됨 | 대시·원생 화면에서의 진입이 한 번에 안 보임, 순서/역할 등 확장 없음, 모바일 터치 피드백 |
| **3번(리포트·결제)** | 리포트 생성·히스토리·한도 노출, 설정 화면 | 결제 실패 복구·해지 전 유지 플로우 없음, 퍼널 가시화 없음 |
| **전역** | 뷰별 기능은 있음 | 온보딩·빈 상태·에러·로딩·재시도 정책이 화면마다 제각각, 에러 바운더리·일관된 피드백 부족 |

## 1단계: 내실 (에러·상태·일관성)

**목표**: 어디서든 **실패가 명확히 보이고, 재시도 가능**하게. 상용화 후 장애 대응 비용을 줄인다.

### 1.1 전역

- **에러 바운더리**  
  - `SpokeduProClient` 또는 레이아웃 한 단계 아래에 Error Boundary 1개.  
  - fallback UI: “일시적인 오류가 났어요. 새로고침해 주세요.” + 새로고침 버튼.  
  - 개발 시 `console.error` 또는 로깅 훅으로 스택 전달(선택).
- **API 재시도·피드백**  
  - 원생/출결/대시보드 fetch 실패 시: 화면에 “다시 시도” 버튼 노출(이미 `syncError` 등 있는 부분은 메시지·버튼 문구 통일).  
  - 재시도 시 로딩 표시(스피너 또는 스켈레톤)로 중복 요청 방지.
- **로딩 일관성**  
  - “데이터 로드 중”일 때: 터치/클릭 비활성화 또는 스켈레톤으로 영역 표시.  
  - 대시보드·원생·리포트·수업 보조 각각 최소 1곳에서 “첫 로드” 상태를 명시적으로 처리.

### 1.2 화면별

- **대시보드(RoadmapView)**  
  - 로드 실패 시: “이번 주 추천을 불러오지 못했어요” + “다시 시도” 버튼.  
  - 빈 상태(아이템 0개): “이번 주 추천이 아직 없어요. 곧 업데이트될 예정이에요.” (운영 정책에 맞게 문구 조정)
- **원생 관리(DataCenterView)**  
  - `syncError` 노출 방식을 토스트 또는 인라인 배너로 통일하고, “다시 시도” 동작을 명확히.
- **AI 리포트(AIReportView)**  
  - 생성 실패 시 이미 에러 메시지 있음 → “다시 생성” 버튼 항상 노출.  
  - 이전 리포트 목록 로드 실패: “목록을 불러오지 못했어요” + 다시 시도.
- **수업 보조(AssistantToolsView)**  
  - 출석 0명일 때 문구는 있음 → “원생 관리에서 출석 처리하기”로 이동하는 링크 또는 버튼 추가 검토.

### 1.3 완료 기준

- [ ] Error Boundary 적용 후 의도적으로 throw 해서 fallback 노출 확인.  
- [ ] 대시/원생/리포트 중 한 경로라도 실패 → “다시 시도”로 복구 가능한지 확인.  
- [ ] 각 메인 뷰에 “로딩 중” 상태가 있어서 빈 화면만 덜어짐.

## 2단계: 1번 고도화 (주간 수업 자료)

**목표**: “매주 4개”가 **신뢰감 있고, 쓰기 쉬운** 주간 가이드라인으로 느껴지게.

### 2.1 데이터·정책

- **주차 기준**  
  - “이번 주”를 **캘린더 주(월요일 시작)** vs **학기 N주차** 중 하나로 고정.  
  - 대시보드 또는 API에 “2025년 3월 3주차” 같은 라벨 노출해 사용자가 “갱신됐다”는 걸 인지하게.
- **144개·프로그램 풀**  
  - “매주 4개” 후보가 **어떤 풀(144개 놀이체육 + 프로그램)**에서 오는지 문서/코드에 명시.  
  - 큐레이션 편집이 그 풀에서만 선택하도록 제한할지, 자유 입력할지 정책 결정.
- **갱신 주기**  
  - 주가 바뀌는 시점(예: 월요일 00:00)에 대시보드 데이터가 바뀌는지 확인.  
  - 필요하면 서버/클라이언트에 “이번 주 키”를 두고, 주가 바뀌면 재fetch하도록.

### 2.2 대시보드 UX

- **헤더/라벨**  
  - Row1 위에 “이번 주 수업 가이드” 또는 “이번 주 추천” 문구를 고정.  
  - 옆에 “3월 3주차” 같은 주차 라벨 노출.
- **카드 액션**  
  - 카드 클릭 시 상세(드로어 등) 외에 “영상 보기” “체크리스트 보기” 등 **실제 수업에서 쓰는 1~2개 액션**을 같은 화면에서 바로 쓸 수 있게.
- **빈/부분 상태**  
  - 4개 미만일 때: “나머지 추천은 곧 채워질 예정이에요” 등으로 불완전함을 설명.  
  - 0개: 1.2의 빈 상태 메시지와 연결.

### 2.3 다음 주·이전 주(선택)

- “다음 주 미리보기” 블록(텍스트만이라도): “다음 주에는 OO 테마 예정” 등으로 기대감.  
- “지난 주 추천” 접기/펼치기: 재사용하고 싶은 코치를 위한 보조 정보.

### 2.4 완료 기준

- [ ] 주차 라벨이 사용자에게 보이고, 주가 바뀌면 내용이 바뀌는 것이 체감됨.  
- [ ] “이번 주 수업 가이드”가 대시보드 첫 블록으로 고정됨.  
- [ ] 144개·프로그램 풀과 “매주 4개” 선정 방식이 문서 또는 코드 주석으로 정리됨.

## 3단계: 2번 고도화 (수업 보조)

**목표**: 술래/팀 나누기가 **수업 직전에 더 빨리 찾아 쓰이고**, 약간의 확장으로 활용도를 높인다.

### 3.1 진입 경로

- **대시보드**  
  - “이번 주 추천” 아래 또는 툴킷 영역에 “오늘 술래·팀 나누기” 같은 **한 줄 CTA** (클릭 시 수업 보조도구 뷰로 전환).  
  - 방향성: 1번 주인공을 해치지 않도록 블록 1개·텍스트 1줄 수준.
- **원생 관리(DataCenterView)**  
  - 출결 저장 후 또는 상단에 “수업 보조도구에서 술래·팀 나누기” 링크/버튼으로 진입 1탭으로 단축.

### 3.2 술래 정하기

- **오늘 술래 제외(선택)**  
  - “이미 뽑은 술래 제외하고 다시 뽑기” 옵션.  
  - 또는 “한 번 더 뽑기” 시 이전에 뽑힌 사람 제외 토글로 구현.
- **모바일**  
  - 버튼 터치 영역 44px 이상, 뽑는 중일 때 햅틱(지원 시) 또는 짧은 진동으로 피드백.

### 3.3 팀 나누기

- **옵션 노출**  
  - “균형 맞추기” / “랜덤” 선택이 한 화면에 명확히 보이게.  
  - 결과 화면에서 “다시 나누기” 시 이전과 다른 조합이 나오는지(랜덤 시드 등) 확인.
- **팀 이름**  
  - A팀/B팀 기본값 유지하되, “이름 바꾸기”로 한 번에 “빨강팀/파랑팀” 등으로 변경 가능하면 가치 있음(우선순위 낮음).

### 3.4 순서 정하기(신규, 소규모)

- 출석 명단 기준으로 **발표 순서** 또는 **번호 부여(1번, 2번…)** 한 번에 생성.  
  - 2번 “완료 정의”(2탭 이내, 한도 없음)를 유지하는 범위에서만 설계.

### 3.5 완료 기준

- [ ] 대시보드 또는 원생 관리에서 “수업 보조”로 1클릭 진입 가능.  
- [ ] 술래/팀 나누기 모바일에서 터치만으로 완료 가능.  
- [ ] (선택) 순서 정하기 1종 추가 후, 2탭 이내에 결과 나오는지 검증.

## 4단계: 3번 고도화 (리포트·결제·리텐션)

**목표**: 리포트는 “꼭 쓸 때만” 유지하고, **결제·해지**로 이탈을 줄인다.

### 4.1 리포트

- **한도·사용량**  
  - “이번 달 N회 남음” 문구가 설정·리포트 화면에 항상 보이게.  
  - 1~2회 남았을 때 경고 배너(이미 비슷한 로직 있음) 문구를 톤 통일.
- **히스토리**  
  - 이전 리포트 목록에 “기간” 라벨이 없으면 `goal` 또는 `created_at`으로 “3월 2주차” 등 표기.  
  - 목록이 길어지면 가상 스크롤 또는 “더 보기”로 부담 줄이기(데이터 100건 제한은 이미 있음).

### 4.2 결제·구독

- **결제 실패**  
  - 결제 실패 시 재시도 링크/버튼을 이메일 또는 앱 내 알림으로 제공.  
  - 설정 화면에 “결제에 문제가 있나요? 다시 시도하기” 진입점.
- **해지 전**  
  - 해지 플로우 진입 시 “일시 정지” “다운그레이드” 옵션 노출.  
  - “다음 결제일까지 사용 가능” 문구로 혼동 방지.

### 4.3 퍼널·가시화(운영)

- **이벤트 정의**  
  - 로그인, 대시보드 조회, “이번 주” 카드 클릭, 수업 보조 도구 사용, 리포트 생성, 설정 조회 등 최소 5~7개.  
  - 추후 분석 툴 연결 시 “주간 활성” “이번 주 추천 노출률” 등 지표 정의.
- **내부 대시**  
  - (선택) 주간 활성 센터 수, 리포트 생성 수, 결제 실패 건수 등을 한 화면에서 볼 수 있게.

### 4.4 완료 기준

- [ ] 결제 실패 시 사용자가 “다시 시도” 경로를 찾을 수 있음.  
- [ ] 해지 전에 일시정지/다운그레이드 중 하나라도 노출됨.  
- [ ] 핵심 이벤트 5개 이상 정의되어 있고, 추후 로깅만 붙이면 퍼널 분석 가능한 상태.

## 5단계: 경험 완성 (온보딩·일관성·접근성)

**목표**: 첫 사용자와 재방문자 모두 **목적을 빨리 이해하고, 실수 시 복구**가 쉬운 앱으로.

### 5.1 온보딩

- **첫 진입**  
  - “센터가 없을 때” 온보딩은 이미 있음.  
  - “이번 주 추천 보기” → “원생 등록” → “수업 보조 써보기” 순서를 1~3단계로 짧게 보여주는지 점검.  
  - 스킵 가능, “다시 보지 않기” 저장.
- **퍼스트 타임 힌트**  
  - 대시보드 첫 방문 시 “여기서 매주 4개 추천을 확인해 보세요” 툴팁 1회(선택).  
  - 수업 보조 첫 방문 시 “출석한 원생이 있으면 여기서 술래·팀을 정할 수 있어요” 1줄 안내.

### 5.2 빈 상태·에러 문구 통일

- **용어**  
  - “원생” / “수강생” / “학생” 중 하나로 통일.  
  - “이번 주 추천” / “주간 가이드” 중 하나로 통일.
- **빈 상태**  
  - 모든 목록/리스트에 “아무것도 없을 때” 문구 + (가능하면) 다음 액션 1개(예: “원생 추가하기”).
- **에러**  
  - “문제가 발생했어요. 잠시 후 다시 시도해 주세요.” 등 짧은 문구 + “다시 시도” 버튼 패턴을 공통으로.

### 5.3 접근성·모바일

- **포커스·키보드**  
  - 주요 버튼·탭이 키보드 포커스 가능한지, 포커스 링이 보이는지 확인.  
  - 모달/드로어 열렸을 때 포커스 트랩.
- **모바일**  
  - 터치 타겟 44px 이상, 스와이프로 탭 전환(수업 보조 등)은 선택.  
  - 세로 모드에서 대시보드 4개 카드가 2x2로 읽기 좋은지 확인.

### 5.4 완료 기준

- [ ] 온보딩 스킵·완료 후 재진입 시 다시 안 뜨는지 확인.  
- [ ] “원생/수강생/학생” 중 하나로 문서·화면 문구 통일.  
- [ ] 주요 플로우 1개 이상을 키보드만으로 진행 가능.  
- [ ] 모바일에서 대시·원생·수업 보조 각각 1회 이상 터치로 완료 테스트.
